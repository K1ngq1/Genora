# AI 生成画布项目

## 项目简介

本项目是一个基于可视化画布的 AI 图片与视频生成平台。用户可以在前端画布中输入提示词、上传参考图片、选择生成比例、清晰度和模型类型，然后由后端统一创建任务、调用外部模型 API、处理任务状态、保存生成结果，并将图片或视频返回给前端展示。

项目的核心目标是将多个 AI 模型能力整合到一个统一的画布工作流中，让用户可以更直观地完成文生图、图生图、文生视频、图生视频等生成任务。

---

## 项目定位

本项目不是普通的单页生成工具，而是一个面向多模型、多任务、多流程的 AI 创作画布系统。

它主要用于解决以下问题：

* 通过统一后端接入不同的 AI 模型 API
* 让用户通过画布完成图片和视频生成流程
* 统一管理生成任务的创建、排队、生成、失败、超时和完成状态
* 统一处理用户上传图片、生成图片、生成视频等文件资源
* 为后续用户系统、额度系统、支付系统和商业化部署预留扩展空间

## 核心功能

* 可视化画布生成工作流

## 基础生成流程

### 图片生成流程

```txt
用户输入 prompt
        ↓
前端提交生成参数
        ↓
后端创建生成任务
        ↓
后端调用图片模型 API
        ↓
模型返回图片结果
        ↓
后端保存图片结果
        ↓
前端展示生成图片
```

### 视频生成流程

```txt
用户输入 prompt 或上传参考图片
        ↓
前端提交视频生成请求
        ↓
后端创建本地任务
        ↓
后端调用视频模型 API
        ↓
上游返回 task_id
        ↓
后端保存上游任务 ID
        ↓
前端或后端轮询任务状态
        ↓
任务完成后获取视频地址
        ↓
前端展示生成视频
```

## 系统架构

项目整体分为四层：

```txt
前端画布层
   ↓
后端任务服务层
   ↓
模型 Provider 适配层
   ↓
数据库与文件存储层
```

### 前端画布层

前端主要负责用户交互和结果展示，包括：

* 收集用户输入
* 上传参考图片
* 提交生成任务
* 展示任务状态
* 展示图片和视频结果
* 管理画布节点状态

### 后端任务服务层

后端主要负责统一管理生成任务，包括：

* 校验请求参数
* 创建任务记录
* 调用模型服务
* 保存上游任务 ID
* 更新任务状态
* 处理失败和超时
* 返回标准化结果给前端

### 模型 Provider 适配层

不同模型供应商的 API 参数、返回格式和轮询方式可能不同，因此项目通过 Provider Adapter 统一封装模型调用逻辑。

每个 Provider 主要负责：

* API Key 管理
* 请求参数转换
* 图片比例和尺寸映射
* 正向 / 反向提示词处理
* 图片 URL 处理
* 上游任务 ID 解析
* 生成结果解析
* 错误信息标准化

### 数据库与文件存储层

数据库用于保存任务、用户、模型、文件和生成记录。

文件存储用于保存：

* 用户上传图片
* 生成图片
* 生成视频
* 临时文件
* 任务结果文件

## 任务状态说明

项目建议统一使用以下任务状态，方便前端展示和后端维护：

| 状态         | 说明                 |
| ---------- | ------------------ |
| created    | 任务已创建，但还未提交到模型     |
| queued     | 任务已提交，上游模型正在排队     |
| processing | 模型正在生成             |
| completed  | 任务生成完成             |
| failed     | 任务生成失败             |
| timeout    | 本地轮询超时，但上游任务可能仍在继续 |
| cancelled  | 用户取消任务             |

## 图片上传说明

当用户上传本地图片作为参考图时，不能直接将本地文件路径传给模型供应商，因为上游模型无法访问用户电脑中的本地文件。

推荐流程如下：

```txt
用户上传本地图片
        ↓
前端提交图片文件
        ↓
后端接收并校验文件
        ↓
后端上传到 Storage
        ↓
生成公网 URL 或 signed URL
        ↓
后端将图片 URL 传给模型 API
```

这样可以避免上游模型无法读取图片，导致任务长期处于 `queued` 或最终生成失败。

## API 设计示例

### 创建图片生成任务

```txt
POST /api/images/generate
```

请求示例：

```json
{
  "prompt": "A luxury leather belt product photo",
  "negativePrompt": "low quality, blurry",
  "aspectRatio": "16:9",
  "quality": "hd",
  "model": "agnes-image"
}
```

### 创建视频生成任务

```txt
POST /api/videos/generate
```

请求示例：

```json
{
  "prompt": "A cinematic product video of a leather belt",
  "imageUrl": "https://example.com/uploaded-image.png",
  "aspectRatio": "16:9",
  "quality": "hd",
  "model": "agnes-video"
}
```

### 查询任务状态

```txt
GET /api/tasks/:taskId
```

返回示例：

```json
{
  "id": "task_123",
  "status": "completed",
  "outputUrl": "https://example.com/result.mp4",
  "error": null
}
```

### 上传图片

```txt
POST /api/uploads
```

返回示例：

```json
{
  "url": "https://example.com/uploads/image.png"
}
```

## 环境变量

项目中的密钥必须全部放在后端环境变量中，不能暴露到前端。

示例 `.env.example`：

```env
DATABASE_URL=

SUPABASE_URL=
SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=

AGNES_API_KEY=
APIMART_API_KEY=

NEXT_PUBLIC_APP_URL=http://localhost:3000
```

注意事项：

* 不要将真实 `.env` 文件提交到 GitHub
* 模型 API Key 只能在后端使用
* `SUPABASE_SERVICE_ROLE_KEY` 只能在服务端使用
* 前端只能读取 `NEXT_PUBLIC_` 开头的公开变量
* 生产环境需要在部署平台单独配置环境变量

## 安全设计

为了保证项目后续可以稳定上线，项目需要注意以下安全原则：

* 前端不得直接调用模型供应商 API
* 所有 API Key 必须由后端统一管理
* 用户上传文件需要限制大小和类型
* 生成任务需要绑定用户身份
* 不同用户之间的任务和文件不能互相访问
* 生产环境需要加入接口限流
* 生产环境需要记录模型调用日志
* 生成失败时需要保存错误原因，方便排查

正确调用方式：

```txt
前端 → 自己的后端 → 模型供应商
```

错误调用方式：

```txt
前端 → 模型供应商
```

## 常见错误处理

| 错误                | 可能原因            | 处理方式                  |
| ----------------- | --------------- | --------------------- |
| 401 Unauthorized  | API Key 错误或权限不足 | 检查 API Key 和模型权限      |
| 403 Forbidden     | 接口无权限或被阻止       | 检查供应商权限、额度和请求格式       |
| queued timeout    | 上游仍在排队          | 保存 task_id，允许后续继续查询   |
| failed            | 模型生成失败          | 记录错误并返回前端             |
| image unreadable  | 上游无法读取图片        | 使用公网 URL 或 signed URL |
| model at capacity | 模型拥堵            | 提示用户稍后重试或切换模型         |
| invalid ratio     | 比例参数不支持         | 后端统一转换为模型支持格式         |

## 部署建议

推荐部署组合：

```txt
前端：Vercel / Netlify
后端：Vercel API Routes / Railway / Render
数据库：Supabase PostgreSQL
文件存储：Supabase Storage
```

部署前需要确认：

* 环境变量已完整配置
* 数据库已完成迁移
* Storage bucket 已创建
* 用户上传图片可以被模型供应商访问
* 模型 API Key 有正确权限
* 前端请求地址已指向生产后端
* 图片和视频结果可以正常展示

## 维护原则

为了保证项目后续稳定维护，建议遵守以下原则：

* 前端只负责交互和展示，不直接调用模型 API
* 后端统一管理 API Key、任务状态和模型请求

* 所有任务状态必须标准化

* 所有错误必须记录清楚
* 不要把真实密钥写进代码或 README
* README 保持清晰，方便后续开发者和 Codex 理解项目
